The previous page showed dependent types winning one small comparison. This page states the general case, then the costs; both sides have serious arguments.

The case for

1. One mechanism: logic, spaces, structures

A proof of \(\forall x \in \mathbb{R},\, P(x)\) is a function assigning to each \(x\) a proof of \(P(x)\); its type, (x : ℝ) → P x, contains the variable \(x\) — a dependent function type. A proof of \(\exists x,\, P(x)\) is a dependent pair. These are the same two constructions that built \(\mathbb{R}^n\), and they also build structures: “a set with an operation and axioms” is a package in which the types of the later components mention the earlier ones —

structure MyGroup where
  carrier : Type
  op      : carrier → carrier → carrier
  e       : carrier
  assoc   : ∀ a b c, op (op a b) c = op a (op b c)
  …

Quantifiers, \(\mathbb{R}^n\), groups, propositions-as-types: one idea. Andrej Bauer’s MathOverflow answer on why proof assistants favor dependent type theory develops this: mathematical language is saturated with dependency — “a vector space over \(F\)”, “a \(\delta\) depending on \(\varepsilon\)” — and this is the formalism that writes it directly.

2. The checker verifies mathematics, not only spelling

In set theory the checker catches nothing; in HOL, category errors; in Lean also dimensions, domains, and hypotheses, since these now live in types. Because propositions are types, proof checking is the same algorithm, performed by a kernel small enough to have several independent reimplementations, applying the same few typing rules to all of Mathlib and to the exercise of Part 1 alike. This is what makes a proof too large to survey by hand still trustworthy.

3. It scales

Mathlib passes a million lines, reaching current research; the Liquid Tensor Experiment verified a new theorem of Clausen–Scholze at Scholze’s request; a formalization of Fermat’s Last Theorem is under way. For fairness: on Wiedijk’s 100-theorems list HOL Light and Isabelle remain slightly ahead of Lean, and Mizar’s set-theoretic library dates to 1973 — every serious foundation has formalized serious mathematics. The defensible claim is not that dependent types are uniquely capable, but that the formal statements coincide with the intended ones, at scale.

The price

Each cost below is the same move — mathematics inside the grammar — seen from the other side. (Two structural limits, universes and proof strength, are on the next page.)

1. Two notions of equality

Equalities that hold by unfolding definitions — \((g \circ f)(x) = g(f(x))\) in Part 1 — cost nothing (definitional). All others, such as \(n + m = m + n\), are theorems (propositional). The distinction has no counterpart on paper; Buzzard avoids teaching it early, calling definitional equality “a non-mathematical concept” — but the machine imposes it, and one meets it the first time a goal refuses to close although both sides are “obviously” equal.

2. Types that are equal but not the same

The consequence flagged in the previous page’s comparison: Fin (n + m) → ℝ and Fin (m + n) → ℝ are provably equal yet distinct types, so an element of one must be transported along the proof of \(n+m = m+n\) to be used as the other. Paulson’s critique “Why not just use Lean?” leads with this example, and it is not a caricature: in Mathlib, associativity of concatenation of tuples reads

Fin.append (Fin.append a b) c = Fin.append a (Fin.append b c) ∘ Fin.cast ⋯

— the Fin.cast is a transport needed merely to state the law. Rewriting under dependencies fails often enough to have its own error message (“motive is not type correct”) and dedicated tactics; the community’s name for the phenomenon, “dependent type hell”, indicates how routinely it is felt.

3. Checking is heavier

Deciding definitional equality means evaluating inside types, so type checking is undecidable in general and slow in practice — Paulson again, and a real contrast with HOL, whose checking is simple and fast by construction.

4. Totality reintroduces junk

Lean’s functions are total, so Mathlib sets \(x/0 = 0\): division returns a junk value outside its mathematical domain, and theorems carry \(b \neq 0\) hypotheses (Buzzard’s FAQ explains why this is sound). Note the symmetry: set theory’s junk was expelled from the grammar and returned through totality. No foundation eliminates convention; each chooses where to keep it.

5. Practitioners economize on them

Mathlib itself uses dependent types sparingly where data is computed with: its matrices are the plain function type Matrix m n α := m → n → α, and its vectors for mathematics, List.Vector α n, are lists bundled with a propositional length proof — structurally the HOL solution, chosen by people who had dependent types available, to avoid transport in computational lemmas. Dependent types earn their keep in statements (they are the quantifiers) and in indexing; Paulson’s formulation — “the secret of dependent types is knowing when not to use them” (2025) — matches Mathlib’s observed practice.

Standing of the question

The Mathlib position paper (CPP 2020) holds that at scale “judicious use of dependent types … [is] unavoidable”; Paulson answers with Grothendieck schemes formalized in dependent-type-free Isabelle (Bordg–Paulson–Li 2022) that “we never hit any wall”. Both claims are supported by the libraries behind them; the choice is an engineering tradeoff, not a theorem. What can be stated precisely is where the tradeoff becomes a hard boundary — next page.
